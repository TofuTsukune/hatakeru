## フェーズ2：畑マップへの日付表示追加

### 要求

> 畑マップに植え付け日と収穫予定を表示してほしいです

### 実装内容

1. セルの高さを 68px → 92px に拡張
2. `fmtShort()` ヘルパーを追加（`2026-06-12` → `6/12` に短縮）
3. セル内に `植:M/D`・`収:M/D` を表示

```js
function fmtShort(d) { if (!d) return null; const [,m,day] = d.split('-'); return `${+m}/${+day}`; }
```

### 知見

- セルサイズは情報量に合わせて調整。小さなセルに詰め込む場合は `font-size: 0.55rem` 程度まで下げて可読性を維持
- 日付は `YYYY-MM-DD` で保持し、表示時だけ短縮すると管理が楽

## フェーズ3：マップから直接作物編集

### 要求

> 作物の情報は畑マップから直接編集したいです

### 実装内容

セルクリック時のモーダルに作物編集フォームを統合した。

**変更前:** セルクリック → 作物を選ぶだけ  
**変更後:** セルクリック → 作物選択 ＋ 名前・品種・日付・ステータス・メモを編集

```js
function updateCellCropEdit(keepDates) {
  // 選択中の作物の情報をフォームに展開する
  // keepDates=true: 既存の区画日付を維持
  // keepDates=false: 作物切り替え時は日付クリア
}
```

### 知見：モーダル拡張パターン

- 既存モーダルに「条件付き展開セクション」を追加するパターンは再利用性が高い
- 作物選択ドロップダウンの `change` イベントでフォームを動的に出し入れする
- 別モーダルへのジャンプよりも「1モーダルで完結」の方がUXが良い場合がある

## フェーズ4：データ構造の設計変更（最重要）

### 要求

> 植え付け日・収穫日の設定単位は作物単位ではなく、個別に設定させてください。
> A2にトマトを6/12に植え付けるが、A3に別のトマトを6/13に植え付ける日もあるわけです

### 問題

初期設計では `plantedDate`・`expectedHarvest` を**作物オブジェクト**に持っていた。  
同じ作物（トマト）を複数区画に植えた場合、日付が上書きされてしまう。

### データ構造の変遷

**変更前:**
```json
// st.plots
{ "0-1": "cropId文字列" }

// st.crops[n]
{ "id": "...", "name": "トマト", "plantedDate": "2026-06-12", ... }
```

**変更後:**
```json
// st.plots（区画ごとに日付を持つ）
{ "0-1": { "cropId": "...", "plantedDate": "2026-06-12", "expectedHarvest": "2026-08-15" } }

// st.crops[n]（日付フィールドを削除）
{ "id": "...", "name": "トマト", ... }
```

### 後方互換性の処理

既存データが文字列形式（旧フォーマット）の場合に対応するコードを追加:

```js
const plotData = st.plots[key];
const cropId = typeof plotData === 'object' ? plotData?.cropId : plotData;
```

### 影響範囲

| 箇所 | 変更内容 |
|------|---------|
| `renderMap()` | `plots[key]` をオブジェクトとして読む |
| 凡例 | `usedIds` の抽出ロジックを修正 |
| `openCellModal()` | 区画日付を `activePlotDates` 変数で保持 |
| `cell-save` | `plots[key]` にオブジェクトとして保存 |
| 作物削除 | cropId の比較ロジックを修正 |
| `renderCalendar()` | `st.crops` ではなく `st.plots` を走査 |
| 作物モーダル | 日付入力欄を削除 |
| 作物カード | 日付表示を削除 |

### カレンダーへの反映

区画ごとの日付をカレンダーに表示する際、同じ作物が複数区画にある場合を区別するため**区画番号**を付加:

```
🌱 植付け: トマト (A2)
🌱 植付け: トマト (A3)
```

```js
Object.entries(st.plots).forEach(([pkey, plotData]) => {
  const [pr, pc] = pkey.split('-').map(Number);
  const lbl = cellLbl(pr, pc);  // A2, A3 など
  // ...
});
```

### 知見：データ設計の原則

> **「何が1対多の関係か」を最初に見極める**

- 作物とセルは多対多（同じ作物を複数セルに配置できる）
- 日付は「植えたイベント」に属する情報 → セル（区画）に持つべき
- 「〇〇単位で設定したい」というユーザー要求はデータの粒度を見直すサイン

**アンチパターン（今回踏んだ失敗）:**
```
作物マスタに日付を持つ → 同じ作物を複数区画に植えると日付が衝突する
```

## 技術的知見まとめ

### 単一HTMLファイルアプリの設計指針

- CSS変数（`--bg`, `--surface` 等）でダークテーマを一元管理
- `localStorage` 1キーにJSONまるごと保存するシンプルな永続化
- `uid()` = `Date.now().toString(36) + random` でサーバー不要のID生成
- `esc()` 関数で表示テキストを必ずサニタイズ（XSS対策）

### UI設計パターン

```
タブ切り替え → モーダル → 一覧再描画
```
- 各タブに `renderXxx()` 関数を1つ持ち、保存のたびに再描画する
- モーダルの開閉は `hidden` クラスの付け外しだけで制御
- オーバーレイ背景クリックで閉じる操作感はユーザーに自然

### 反省点・次回への改善案

1. **データ設計は最初に固める** — 要件定義時に「同じ作物を複数区画に植えますか？」と確認できていればフェーズ4の大規模変更は不要だった
2. **後方互換コードの蓄積** — `typeof plotData === 'object'` のような分岐は技術的負債になりやすい。初期データマイグレーション関数を用意する方が清潔
3. **モーダルの役割分離** — セルモーダルに作物編集まで統合したことで役割が広がりすぎた面もある。大規模化する場合は分離を検討
